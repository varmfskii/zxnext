TSconf: Ports

Development
In this section, I would like to describe all the port groups involved in the management of the TSconf system.
This section describes the general purpose of ports. The complete dock for the ports lies here .

So, the system ports can be divided into the following groups:
  • control ports for code execution by the processor. This includes such ports as: SysConfig, CacheConfig
  • memory management ports: MemConfig, Page0 - Page3, FMAddr
  • TSU graphics page ports: VPage, T0GPage, T1GPage, TMPage, SGPage
  • Graphics / Color Management ports: VConfig, PalSel, Border
  • display control ports: TSConfig, GXOffs / GYOffs, T0XOffs / T0YOffs, T1XOffs / T1YOffs
  • DMA management ports: DMASAddr, DMADAddr, DMALen, DMANum, DMACtrl
  • ports of management of arrival INT: INTMask, HSINT, VSINT
  • Virtual FDD Management Port: FDDVirt

Tao says: Configuration management occurs through the programming of ports. The system ports can also be programmed by writing data to FMaps from address # 0400.
There are two types of ports according to the way data is specified: changing individual significant bits, or specifying 8-bit values.
Data for some devices is collected from a pair of ports, in the names of which the letters H (high byne) & L (low byte) appear.

So, consider in order:

Management ports for code execution by the processor. SysConfig, CacheConfig
The SysConfig port (# 20af) is designed to control the speed of the z80, allowing you to turn on the processor at speeds of 3.5, 7.0 and 14.0 MHz, and also allows you to enable the internal cache of the system.
Cache usage can be controlled by the CacheConfig port (# 2baf) , which allows you to enable cache usage for all memory windows: EN_C000 EN_8000 EN_4000 EN_0000

System Cache Usage
The cache saves processor access to RAM. With each reading, the bytes read from the RAM and the adjacent bytes are copied into the cache, and all the following references will be read from it. The performance gain is achieved by the absence of processor braking, which occurs when accessing physical. RAM, as well as saving RAM cycles, which are available for other devices.
Example 1 (adjacent addresses):
- 0x8000 address is read, RAM is accessed, data from 0x8000 and 0x8001
is cached - address is 0x8001, cache is accessed

Example 2 (non-adjacent addresses):
- address is read 0x8001, RAM is accessed, cache data from addresses 0x8000 and 0x8001
- address 0x8002 is read, access to RAM, data from addresses 0x8002 and 0x8003 get to the cache

The cache size is 512 bytes , but the data memory is located in such a way that it coincides with the lower nine bits of the processor address. Therefore, successive accesses to data located at addresses whose lower 9 bits match will cause a conflict and overrun of previous data from the cache. Such a cache is called a direct mapping cache and is most easily implemented for FPGA.

Example 1 (conflicting addresses):
- 0x8000 address is read, RAM is accessed, data from 0x8000 and 0x8001
is cached - address is 0x8200, address is to RAM, data from 0x8200 and 0x8201 addresses are in the cache (data from 0x8000 and 0x8001 addresses are lost)

Example 2 (non-conflicting addresses):
- read the address 0x8000, an appeal to the RAM, the cache get data from 0x8000 addresses and 0h8001
- read address 0h8100, appeal to the RAM, the cache get data from addresses 0h8100 and 0h8101 (data address 0x8000 and 0h8001 saved)

From this we can conclude that for more rational use of the cache, it is advisable to place the code on "non-intersecting" addresses. For example, if the use of a string inta is required. This is not very convenient, but they resorted to other tricks on other platforms.

The cache takes into account page addressing , so the data at 0xC000 on page 0 and the data at 0xC000 on page 1 will be different.

Example
- page 0 is turned on, the address is 0xС000, the address to the RAM is read, data from the addresses 0xC000 and 0xC001 are in the cache
- page 1 is turned on, the address is 0xС000, the address to the RAM is read, data from the addresses 0xC000 and 0xC001
is recalculated - page 1 is turned on, the address is 0xС001, the address to the cache is read

Ports of memory management. MemConfig, Page0 - Page3, FMAddr
Port # 21af - MemConfig is described here .
Ports Page0 - Page3 (# 10af - # 13af) are designed to display pages in the specified RAM address windows: # 0000, # 4000, # 8000, # c000, which covers the entire address space of the memory.
The FMAddr port (# 15af) is designed to control the reflection of the system's internal memory in the memory available to the processor for recording.
We have the ability to enable this window at multiple # 1000 addresses.
The following data is stored in the internal memory of the system: a palette is stored at relative addresses 0-511 , and sprites handles are stored at addresses 512-1023, and from the address # 400 there is a port mapping block. Writing to these addresses leads to programming of system ports, like direct writing to a port.

Ports of TSU graphics pages. VPage, T0GPage, T1GPage, TMPage, SGPage
These ports are designed to specify the pages used by the TSU (tile-sprite engine that displays everything on the screen) to display the video, as well as graphics of tile layers and sprites.
The VPage port (# 01af) indicates to the system, starting from which page a consecutive set of memory pages will be used as video pages .
Ports T0GPage (# 17af) , T1GPage (# 18af), SGPage (# 19af) point to the pages that contain the prepared tile graphics for output. TMPage
port (# 16af) points to the page used as a tile map.

Ports control graphics modes / color. VConfig, PalSel, Border
The VConfig port (# 00af) is intended for the following tasks: setting the resolution and video mode, as well as controlling the display of the TSU and graphics layer.
The PalSel port (# 07af) is used to select a palette that will be used in the main graphic screen, and also allows you to specify the upper two bits of the palette number for tile layers. Combining the two high bits of PalSel for the tiles, and the selected lower two bits of the palette in the tile descriptor, we get the number of the specific tile palette.
Port Border (# 0faf) indicates the color of the border in the current system palette.

Display control ports. TSConfig, GXOffs / GYOffs, T0XOffs / T0YOffs, T1XOffs / T1YOffs
The TSConfig port (# 06af) is intended for TSU and allows you to enable / disable the display of sprites, tile layers by setting the S_EN, T1_EN, T0_EN bits to 1. Bits T0Z_EN and T1Z_EN are designed to enable the display of tiles number 0. The

set of ports X_Offs is used to control the window display the specified resolution for the corresponding port layer - a graphic screen or tile layers.
Since the video memory field (and the display of tiles) is 512 pixels wide and looped, we use two ports (major and minor) to control the display window in order to specify 9 data bits, which allow to specify all available positions 0-511.
Example: T0XOffsL - low 8 bits, T0XOffsH - high bit.

Ports of the position of the window are a pair for X and a pair for Y:
Graphics: # 02af - GXOffsL, # 03af - GXOffsH; # 04af - GYOffsL, # 05af - GYOffsH.
Tile layer 0: # 40af - T0XOffsL, # 41af - T0XOffsH; # 42af - T0YOffsL, # 43af - T0YOffsH.
Tile layer 1: # 44af - T1XOffsL, # 45af - T1XOffsH; # 46af - T1YOffsL, # 47af - T1YOffsH.

DMA management ports. DMASAddr, DMADAddr, DMALen, DMANum, DMACtrl
Ports indicate the address of the source, the receiver, their pages, the length of the transmitted data and the number of such transactions, as well as the transmission control port.
These ports are sensibly described in the relevant article of the DMA
DMASAddrL (# 1aaf) , DMASAddrH (# 1baf) , DMASAddrX (# 1caf)
DMADAddrL (# 1daf) , DMADAddrH (# 1eaf) , DMADAddrX (# 1faf)

DMALen (# 26af) , DMANum (# 28af)
DMACtrl / DMAStatus (# 27af) - management port, intended both for recording and for reading DMA status.

Ports control ports INT. INTMask, HSINT, VSINT
The INTMask port (# 2aaf) allows you to control interrupt handlers.
The HSINT port (# 22af) indicates the horizontal point of arrival of the interrupt, and the VSINT port pair is the high VSINTH (# 24af) (the low bit is significant) and the low VSINTL (# 23a4) (8). bit) - vertical position (string).
More about intah - here

Virtual drive management. Fddvirt
The FDDVirt port (# 29af) uses 4 low-order bits, each of which indicates all available A-D drives.
The drive operates under these conditions:
- the desired drive bit = 1.
- sootv. flop in FF port [1: 0]
- enabled trdos
- any betadisc port was accessed
This turns on pagi 255 at 0-3ffff
any call to any betadisc port causes pag 255 to be turned off
